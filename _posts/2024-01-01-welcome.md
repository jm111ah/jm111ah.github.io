## 이 블로그를 만든 과정

이 블로그는 AI와의 협업을 통해 처음부터 만들어졌습니다. 전체 과정을 정리해보겠습니다.

### 1. Git 저장소 연결

GitHub 저장소(`jm111ah.github.io`)를 로컬 디렉토리(`D:\Memory\Git\Blog`)에 연결하는 것부터 시작했습니다.

### 2. Jekyll 블로그 초기 세팅

Jekyll을 사용한 정적 사이트 생성 블로그로 설정했습니다:
- `_config.yml`: 블로그 설정 파일
- `Gemfile`: Ruby 의존성 관리
- `_layouts/`: HTML 레이아웃 템플릿
- `_posts/`: 블로그 포스트 디렉토리
- `assets/css/`: 스타일시트
- GitHub Actions 워크플로우 설정

#### Jekyll이란?

**Jekyll**은 정적 사이트 생성기(Static Site Generator)입니다. Ruby로 작성되었으며, 마크다운 파일과 템플릿을 사용하여 완전한 정적 웹사이트를 생성합니다.

**주요 특징:**

1. **정적 사이트 생성**: 데이터베이스 없이 HTML, CSS, JavaScript로만 구성된 정적 사이트를 생성합니다.
2. **마크다운 지원**: 포스트를 마크다운 형식으로 작성하면 자동으로 HTML로 변환됩니다.
3. **템플릿 시스템**: Liquid 템플릿 언어를 사용하여 재사용 가능한 레이아웃을 만들 수 있습니다.
4. **GitHub Pages 통합**: GitHub Pages에서 Jekyll을 기본적으로 지원하여 무료 호스팅이 가능합니다.
5. **빠른 로딩 속도**: 정적 파일만 제공하므로 매우 빠른 로딩 속도를 제공합니다.

**Jekyll의 작동 방식:**

1. 마크다운 파일(`.md`)과 템플릿 파일을 작성합니다.
2. Jekyll이 이를 처리하여 완전한 HTML 사이트를 생성합니다.
3. 생성된 정적 파일을 웹 서버에 배포합니다.

**Jekyll의 주요 구성 요소:**

- **Front Matter**: 파일 상단에 있는 YAML 형식의 메타데이터 (제목, 날짜, 카테고리 등)
- **Layouts**: 페이지의 기본 구조를 정의하는 템플릿
- **Includes**: 재사용 가능한 HTML 조각
- **Posts**: `_posts` 폴더에 저장되는 블로그 포스트
- **Pages**: 일반 웹 페이지
- **Assets**: CSS, JavaScript, 이미지 등 정적 파일

**Jekyll의 장점:**

- ✅ 무료 호스팅 (GitHub Pages)
- ✅ 버전 관리 (Git과 통합)
- ✅ 빠른 로딩 속도
- ✅ SEO 친화적
- ✅ 보안 (데이터베이스 없음)
- ✅ 마크다운으로 쉽게 작성 가능

이러한 특징들로 인해 많은 개발자들이 기술 블로그를 운영할 때 Jekyll을 선택합니다.

### 3. Git 설치 및 설정

Windows 환경에서 Git이 설치되어 있지 않아 자동으로 다운로드하고 설치했습니다:
- Git 설치 파일 다운로드
- 자동 설치 진행
- Git 사용자 정보 설정 (이름, 이메일)

### 4. GitHub 인증 설정

GitHub에 push하기 위해 Personal Access Token이 필요했습니다:
- GitHub Personal Access Token 생성
- 토큰 권한 설정 (`repo` 권한 필요)
- 원격 저장소 URL에 토큰 포함하여 인증

### 5. 첫 Push

여러 시행착오를 거쳐 성공적으로 push했습니다:
- Git 저장소 초기화
- 파일 추가 및 커밋
- 원격 저장소와의 충돌 해결 (force push)
- 최종적으로 성공적으로 업로드 완료

### 6. 포스팅 추가 방법 학습

블로그 업데이트 프로세스를 학습했습니다:
- `_posts` 폴더에 날짜 형식의 마크다운 파일 생성
- Jekyll front matter 작성
- Git으로 커밋 및 push
- GitHub Actions를 통한 자동 배포

### 주요 학습 내용

1. **Jekyll 블로그 구조**: 정적 사이트 생성기의 기본 구조와 작동 방식
2. **Git 워크플로우**: add → commit → push의 기본 프로세스
3. **GitHub 인증**: Personal Access Token을 사용한 인증 방법
4. **자동 배포**: GitHub Actions를 통한 CI/CD 파이프라인

- 자동화 도구의 편리함

을 경험할 수 있었습니다.
