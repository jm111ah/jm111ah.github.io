## 프로젝트 개요

GitHub Pages와 Jekyll을 활용하여 정적 사이트 생성 기술 블로그를 구축했습니다. 이 프로젝트는 마크다운 기반의 콘텐츠 작성과 자동화된 배포 프로세스를 통해 효율적인 블로그 운영을 목표로 합니다.

이 블로그는 **Cursor IDE**를 사용하여 처음부터 구축했습니다. AI 기반 코드 에디터인 Cursor를 통해 Jekyll 블로그의 초기 세팅부터 배포까지 전체 과정을 진행하면서, 개발 도구의 효율성과 AI 협업의 가능성을 직접 경험할 수 있었습니다.

## 기술 스택

- **Jekyll**: 정적 사이트 생성기 (Ruby 기반)
- **GitHub Pages**: 무료 웹 호스팅 서비스
- **Git**: 버전 관리 시스템
- **Markdown**: 콘텐츠 작성 언어
- **Liquid**: 템플릿 엔진
- **HTML/CSS**: 프론트엔드 구조 및 스타일링
- **Cursor IDE**: AI 기반 코드 에디터 (개발 도구)

## 프로젝트 구조

```
Blog/
├── _config.yml          # Jekyll 설정 파일
├── Gemfile              # Ruby 의존성 관리
├── _layouts/            # HTML 레이아웃 템플릿
│   ├── default.html     # 기본 레이아웃
│   └── post.html        # 포스트 레이아웃
├── _posts/              # 블로그 포스트 디렉토리
├── assets/              # 정적 파일 (CSS, 이미지 등)
│   └── css/
│       └── main.css     # 커스텀 스타일시트
├── index.html           # 메인 페이지
├── blog.html            # 블로그 목록 페이지
└── about.md             # 소개 페이지
```

## 개발 과정 및 경험

### Cursor IDE를 선택한 이유

이 프로젝트를 시작할 때, Jekyll 블로그에 대한 경험이 전혀 없었습니다. 처음에는 어떻게 시작해야 할지 막막했는데, Cursor IDE의 AI 기능을 활용하면 단계별로 가이드를 받을 수 있을 것 같아 선택하게 되었습니다.

### Cursor IDE를 통한 개발 경험

Cursor IDE를 사용하면서 가장 인상 깊었던 점은:

1. **대화형 개발**: "Jekyll 블로그 초기 세팅을 해줘"라고 요청하면 필요한 파일 구조와 설정 파일들을 자동으로 생성해주었습니다. 단순히 코드만 생성하는 것이 아니라, 프로젝트의 전체 구조를 이해하고 적절한 파일들을 만들어주었습니다.

2. **문제 해결 지원**: Git 설치나 GitHub 인증 문제가 발생했을 때, 에러 메시지를 공유하면 원인을 분석하고 해결 방법을 제시해주었습니다. 특히 GitHub Personal Access Token 권한 문제를 해결할 때 큰 도움이 되었습니다.

3. **컨텍스트 이해**: 이전 대화 내용을 기억하고 있어서, "이전에 만든 블로그에 AI 포스팅을 추가해줘"라고 요청하면 블로그 구조를 파악하고 적절한 위치에 파일을 생성해주었습니다.

4. **자동화된 워크플로우**: Git 커밋과 push까지 자동으로 처리해주어, 개발에만 집중할 수 있었습니다.

### 프로젝트 시작 계기

기술 블로그를 운영하고 싶었지만, 기존 블로그 플랫폼의 제약이 있었습니다. GitHub Pages와 Jekyll을 선택한 이유는:

- 무료 호스팅으로 운영 비용 부담 없음
- Git을 통한 버전 관리로 콘텐츠 관리 용이
- 마크다운 기반의 간편한 작성 방식
- 완전한 커스터마이징 가능
- 개발자로서 GitHub를 활용한 프로젝트 경험을 쌓고 싶었음

## 기술적 도전과 해결

### 1. Git 설치 및 환경 설정

**문제 상황**: 
Windows PowerShell에서 `git` 명령어를 실행했을 때 "명령을 찾을 수 없습니다" 오류가 발생했습니다. Git이 시스템에 설치되어 있지 않았던 것입니다.

**해결 과정**:
1. Cursor IDE를 통해 Git 설치 파일을 자동으로 다운로드
2. PowerShell 스크립트를 작성하여 자동 설치 진행 (`/VERYSILENT`, `/NORESTART` 옵션 사용)
3. 설치 후 PATH에 Git 경로 추가 (`C:\Program Files\Git\bin`)
4. Git 사용자 정보 설정 (`git config user.name`, `git config user.email`)
5. 원격 저장소 연결 및 초기화

**배운 점**: 
Windows 환경에서 개발 도구를 자동으로 설치하고 설정하는 방법을 학습했습니다. 특히 PowerShell을 통한 자동화 스크립트 작성 경험이 유용했습니다.

### 2. GitHub 인증 설정 - 가장 큰 도전

**문제 상황**: 
GitHub에 push를 시도했을 때 403 Forbidden 오류가 계속 발생했습니다. 처음에는 토큰이 잘못 생성된 줄 알았지만, 실제로는 권한 설정 문제였습니다.

**시행착오 과정**:
1. **첫 번째 시도**: Personal Access Token을 생성했지만, "Repository access"에서 "Public repositories" (읽기 전용)만 선택하여 push 실패
2. **두 번째 시도**: 토큰 권한을 수정했지만, 원격 저장소 URL에 토큰이 포함되어 있지 않아 여전히 실패
3. **세 번째 시도**: 원격 저장소 URL을 `https://username:token@github.com/...` 형식으로 변경했지만, 토큰 형식 문제로 실패
4. **최종 해결**: 
   - 토큰 권한을 "All repositories"로 변경
   - 원격 저장소 URL을 `https://jm111ah:github_pat_...@github.com/jm111ah/jm111ah.github.io.git` 형식으로 설정
   - `git push -u origin main --force` 명령으로 성공

**배운 점**: 
- GitHub Personal Access Token의 권한 설정이 매우 중요하다는 것을 깨달았습니다
- 읽기 권한과 쓰기 권한의 차이를 명확히 이해하게 되었습니다
- 인증 실패 시 에러 메시지를 자세히 분석하는 것이 중요합니다

### 3. 원격 저장소와의 충돌 해결

**문제 상황**: 
로컬에서 커밋한 내용을 push하려고 했을 때 "non-fast-forward" 오류가 발생했습니다. 원격 저장소에 로컬에 없는 커밋이 있었던 것입니다.

**해결 과정**:
1. `git fetch origin`으로 원격 저장소의 변경사항 확인
2. 원격 저장소의 내용이 로컬과 다르다는 것을 확인
3. 로컬 변경사항을 우선하기 위해 `git push --force` 사용
4. 성공적으로 push 완료

**배운 점**: 
Git의 브랜치 병합 전략과 force push의 위험성에 대해 학습했습니다. 프로덕션 환경에서는 force push를 신중하게 사용해야 한다는 것을 알게 되었습니다.

### 4. Jekyll 설정 최적화

**구현 내용**:
- `_config.yml`에서 사이트 메타데이터 관리 (제목, 설명, 작성자 정보)
- Kramdown 마크다운 파서 설정
- Rouge 코드 하이라이터 설정
- 플러그인 설정 (jekyll-feed, jekyll-sitemap)
- `.gitignore` 파일 설정으로 불필요한 파일 제외

**경험**: 
Jekyll의 설정 파일 구조를 이해하고, 각 설정값이 어떤 역할을 하는지 학습했습니다. 특히 Front Matter의 중요성을 깨달았습니다.

### 5. 프로젝트 구조 설계

**고려 사항**:
- Jekyll 컨벤션 준수 (`_posts`, `_layouts`, `_config.yml` 등)
- 확장 가능한 디렉토리 구조
- 정적 파일 관리 최적화
- 버전 관리에 포함할 파일과 제외할 파일 구분

## 구현한 주요 기능

### 1. Jekyll 기반 정적 사이트 생성

Jekyll을 사용하여 마크다운 파일을 자동으로 HTML로 변환하는 정적 사이트를 구축했습니다.

**주요 구성 요소:**
- **Front Matter**: YAML 형식의 메타데이터 (제목, 날짜, 카테고리, 태그)
- **Layouts**: 재사용 가능한 HTML 템플릿 (`default.html`, `post.html`)
- **Posts**: 날짜 형식 파일명을 가진 마크다운 포스트 (`YYYY-MM-DD-제목.md`)
- **Pages**: 일반 웹 페이지 (`index.html`, `about.md`)

### 2. GitHub Pages 자동 배포

GitHub 저장소에 push하면 자동으로 GitHub Pages에 배포되는 워크플로우를 구축했습니다.

**배포 프로세스:**
1. Cursor IDE에서 마크다운 파일 작성
2. Git으로 커밋 및 push (Cursor를 통해 자동화)
3. GitHub Pages가 자동으로 Jekyll 빌드 및 배포
4. 몇 분 후 https://jm111ah.github.io 에서 확인 가능

### 3. 커스텀 레이아웃 시스템

기본 Jekyll 테마를 확장하여 커스텀 레이아웃을 구현했습니다:
- 포스트 전용 레이아웃 (`post.html`): 제목, 날짜, 카테고리, 태그 표시
- 기본 페이지 레이아웃 (`default.html`): 네비게이션, 헤더, 푸터 포함
- 메타데이터 표시 및 스타일링

### 4. Git 기반 버전 관리

모든 콘텐츠와 설정을 Git으로 관리하여:
- 변경 이력 추적
- 협업 가능한 구조
- 롤백 및 브랜치 관리
- GitHub를 통한 백업

## 개발하면서 느낀 점

이 프로젝트를 진행하면서 몇 가지 인사이트를 얻을 수 있었습니다:

### 1. 단계별 문제 해결의 중요성

Git 설치부터 인증 설정까지 각 단계에서 문제가 발생했지만, 하나씩 해결해나가면서 전체 시스템을 구축할 수 있었습니다. 처음에는 막막했지만, 문제를 작은 단위로 나누어 해결하니 결국 목표를 달성할 수 있었습니다.

### 2. 인증 및 권한 설정의 세심함

GitHub Personal Access Token의 권한 설정을 잘못하여 여러 번 시도해야 했던 경험이, 보안과 권한 관리의 중요성을 일깨워주었습니다. 단순히 토큰을 생성하는 것이 아니라, 어떤 권한이 필요한지 정확히 이해하는 것이 중요하다는 것을 배웠습니다.

### 3. AI 도구의 실용성

Cursor IDE를 사용하면서 반복적인 코드 작성이나 설정 파일 생성 등의 작업을 효율적으로 처리할 수 있었습니다. 특히 인상 깊었던 점은:

- **컨텍스트 이해**: 이전 대화 내용을 기억하고 있어서, 연속적인 작업이 가능했습니다
- **문제 해결 지원**: 에러 메시지를 분석하고 해결 방법을 제시해주어, 디버깅 시간이 크게 단축되었습니다
- **학습 도구**: 단순히 코드를 생성하는 것이 아니라, 각 단계에서 왜 그렇게 해야 하는지 설명해주어 학습에 도움이 되었습니다

AI가 단순히 코드를 생성하는 것이 아니라 개발 과정 전체를 지원할 수 있다는 것을 직접 경험했습니다.

### 4. 문서화의 가치

이 블로그 프로젝트 자체가 하나의 포트폴리오가 되었고, 개발 과정을 기록함으로써 나중에 참고할 수 있는 자료가 되었습니다. 또한 다른 개발자들에게도 도움이 될 수 있다는 점에서 문서화의 가치를 느꼈습니다.
